# SendGrid 製品紹介

Twilio SendGrid は、世界最大級のクラウドベースメール配信プラットフォームです。毎月 **1,000億通以上** のメールを処理し、Uber、Spotify、Airbnb など多くの企業に採用されています。

---

## なぜメール配信は難しいのか？

### 自前のメールサーバーの課題

```
❌ 到達率の低下（スパム扱い）
❌ IPレピュテーションの管理
❌ バウンス処理の実装
❌ スケーリングの難しさ
❌ セキュリティ対策
❌ 法規制への対応
```

### メール配信の裏側

```
送信者 → SendGrid → インターネット → 受信サーバー → 受信箱
                         ↓
                   ┌─────────────┐
                   │ SPF チェック │
                   │ DKIM 検証    │
                   │ DMARC ポリシー│
                   │ コンテンツ分析│
                   │ レピュテーション│
                   └─────────────┘
```

## SendGrid の主要製品

### 1. Email API

プログラムからメールを送信するための API。

| 特徴 | 説明 |
|------|------|
| **RESTful API** | シンプルな HTTP リクエストで送信 |
| **SMTP リレー** | 既存システムからの移行が容易 |
| **SDK** | Node.js, Python, Ruby, PHP, Java, Go, C# 対応 |
| **高速配信** | 数百万通/時間の処理能力 |

```javascript
// たった数行でメール送信
const msg = {
  to: 'user@example.com',
  from: 'noreply@yourapp.com',
  subject: 'Welcome!',
  text: 'Thank you for signing up.'
};
await sgMail.send(msg);
```

**ユースケース**:
- 会員登録の確認メール
- パスワードリセット
- 注文確認・発送通知
- アラート・通知

### 3. Email Validation API

メールアドレスの有効性を事前にチェック。

```javascript
// 送信前にアドレスを検証
const result = await client.validations.email.validate({
  email: 'test@example.com'
});

console.log(result.verdict); // "Valid", "Risky", "Invalid"
```

| 検証項目 | 説明 |
|---------|------|
| **構文チェック** | RFC 準拠の形式か |
| **ドメイン検証** | MX レコードが存在するか |
| **メールボックス確認** | アドレスが実際に存在するか |
| **使い捨てメール検出** | 一時的なメールサービスを検出 |
| **タイポ修正提案** | `gmial.com` → `gmail.com` |

**効果**:
- バウンス率を **最大 98% 削減**
- 送信コストの削減
- レピュテーションの保護

## 到達性を支える技術

### SPF（Sender Policy Framework）

送信元 IP が正規のものか確認する仕組み。

```
DNS TXT レコード:
v=spf1 include:sendgrid.net ~all
```

**意味**: 「sendgrid.net からのメールは正規の送信者です」

---

### DKIM（DomainKeys Identified Mail）

メールが改ざんされていないことを証明する電子署名。

```
DNS TXT レコード:
s1._domainkey.yourdomain.com
→ 公開鍵を設定
```

**仕組み**:
1. SendGrid が秘密鍵でメールに署名
2. 受信サーバーが DNS の公開鍵で検証
3. 一致すれば「改ざんなし」と判定

---

### DMARC（Domain-based Message Authentication）

SPF と DKIM の結果に基づいて、不正メールの扱いを指定。

```
DNS TXT レコード:
v=DMARC1; p=quarantine; rua=mailto:dmarc@yourdomain.com
```

| ポリシー | 動作 |
|---------|------|
| `none` | 監視のみ（レポート収集） |
| `quarantine` | 迷惑メールフォルダへ |
| `reject` | 受信拒否 |

---

### レピュテーション

ISP（Gmail, Yahoo, etc.）が送信者を評価するスコア。

**影響する要素**:

| 要素 | 良い例 | 悪い例 |
|------|--------|--------|
| バウンス率 | < 2% | > 5% |
| 苦情率 | < 0.1% | > 0.5% |
| スパムトラップ | 0 件 | 1件でも危険 |
| エンゲージメント | 高い開封率 | 未開封が多い |

## 日本向け特有の注意点

### 1. キャリアメール対応

docomo, au, SoftBank のキャリアメールは独自のフィルタリングがあります。

| キャリア | 注意点 |
|---------|--------|
| **docomo** | 画像ブロックがデフォルト、HTMLメール注意 |
| **au** | 特定ドメインのブロックリストあり |
| **SoftBank** | 迷惑メール設定が厳しい |

**対策**:
- テキストメールも用意（マルチパート）
- 送信元ドメインの認証を完璧に
- 配信停止リンクを必ず含める

### 2. 特定電子メール法

日本では **特定電子メール法** への準拠が必要です。

```
✅ 必須事項:
- 送信者の氏名・名称
- 送信者の住所
- 苦情・問い合わせ先
- 配信停止の方法
- オプトイン（事前同意）の取得
```

### 3. RFC 非準拠アドレス

日本のキャリアメールには RFC 違反のアドレスが存在する場合があります。

```
例: ..user@docomo.ne.jp（連続ドット）
例: user.@au.com（ドット終わり）
```

SendGrid はこれらのアドレスにも対応していますが、事前の検証をお勧めします。
